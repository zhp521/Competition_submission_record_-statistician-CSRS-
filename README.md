# 比赛提交记录统计器

一个面向竞赛管理开发岗的练手项目。

项目目标是模拟比赛后台对提交记录进行统计分析的核心工作：输入一批提交记录，输出每题提交数、通过数、首次通过时间、每队提交数、通过题数，以及首个 AC 队伍等信息。

当前项目使用 `Python + CSV + 命令行` 实现，重点放在数据建模、统计逻辑拆分和逐步迭代开发上，适合作为学习项目和面试展示项目。

## 项目功能清单

- 使用 `dataclass` 定义提交记录结构

- 统计每题首次通过时间
- 统计每队提交数

- 使用函数拆分读取、统计、输出逻辑

- 输出更接近正式项目的格式化统计结果
- 引入统一的 `analyze_submissions()` 汇总函数

### 计划扩展

- 排行榜生成

## 输入数据格式

当前使用 `CSV` 作为输入格式，表头如下：

```csv
submission_id,team_id,team_name,problem_id,problem_name,status,submit_time
```

字段说明：

- `submission_id`：提交编号
- `team_id`：队伍编号
- `team_name`：队伍名称
- `problem_id`：题目编号
- `problem_name`：题目名称
- `status`：提交结果
- `submit_time`：提交时间

其中：

- `status` 当前支持 `AC`、`WA`、`TLE`、`RE`、`CE`
- `submit_time` 格式为 `YYYY-MM-DD HH:MM:SS`

样例数据位于 `sample_data/submissions.csv`。

## 实现思路

这个项目按照“先读数据，再做最小统计，再逐步扩展”的方式完成。

整体可以拆成 3 层：

1. 数据读取层
- 从 `CSV` 中逐行读取记录
- 将每一行转换成结构化的 `Submission` 对象

2. 业务统计层
- 按题目统计提交次数
- 按题目统计通过次数
- 按时间顺序统计首次通过时间
- 按队伍统计提交次数
- 利用 `set` 去重统计每队通过题数
- 记录每题首个 `AC` 的提交对象

3. 结果展示层
- 将统计结果汇总
- 按更清晰的格式输出到命令行

## 教学式开发步骤

为了方便学习，这个项目采用分阶段手写的方式实现。

### 第一步：定义数据结构

先使用 `@dataclass` 定义一条提交记录：

```python
@dataclass
class Submission:
    submission_id: str
    team_id: str
    team_name: str
    problem_id: str
    problem_name: str
    status: str
    submit_time: datetime
```

这样每条数据就从“字典”变成了“结构清晰的对象”。

### 第二步：读取 CSV

使用 `csv.DictReader` 逐行读取数据，并用 `datetime.strptime()` 把时间字符串转成时间对象。

### 第三步：先做最小统计

从以下功能开始逐步实现：

- 每题提交数
- 每题通过数
- 每题首次通过时间
- 每队提交数
- 每队通过题数

### 第四步：完善展示信息

第二版在基础统计之上，进一步补充：

- 显示题目名称和队伍名称
- 输出每题首个 `AC` 队伍
- 将零散统计整合到统一的分析函数中

## 当前代码结构

当前主程序位于 `main.py`，主要包含以下函数：

- `load_submissions()`：读取提交记录
- `analyze_submissions()`：统一完成题目和队伍统计
- `print_problem_stats()`：输出题目统计
- `print_team_stats()`：输出队伍统计
- `print_first_ac_records()`：输出每题首个 AC 记录

这种拆分方式的好处是：

- 每个函数只做一件事
- 逻辑更清晰
- 更方便后续继续扩展功能

## 第二版功能说明

第二版主要完善的是“更像正式比赛后台”的统计展示能力。

### 1. 题目统计补充题目名称

现在输出不再只有题号，还会显示题目名称，例如：

```text
题目 A (签到题): 提交 3 次, 通过 2 次, 首次通过时间 2026-04-09 09:03:00
```

### 2. 队伍统计补充队伍名称

现在输出不再只有队伍编号，还会显示队伍名称，例如：

```text
队伍 T1 (Alpha): 提交 4 次, 通过记录 2 次, 通过题数 2
```

### 3. 增加每题首个 AC 队伍统计

程序会记录每道题第一次通过的提交记录，并输出对应的队伍和时间。

### 4. 引入统一汇总函数

使用 `analyze_submissions()` 一次遍历完成大部分核心统计，并返回：

- `problem_stats`
- `team_stats`
- `first_ac_records`

这样后续继续扩展排行榜、罚时计算时会更方便。

## 面试时可以重点讲什么

如果把这个项目作为面试项目，可以重点讲下面几点：

- 为什么先选 `CSV`
  适合快速验证业务逻辑，降低起步成本

- 为什么使用 `dataclass`
  让数据结构更清晰，代码更易读

- 为什么先做小统计再扩展
  先保证基础功能正确，再逐步增加复杂业务逻辑

- 为什么统计通过题数时要用 `set`
  因为同一队伍同一题可能多次 `AC`，必须去重

- 为什么首次通过时间需要按时间顺序处理
  因为“首次”本质上就是时间顺序问题

- 为什么第二版要把题目名、队伍名和首个 AC 队伍补上
  因为这会让输出结果更贴近真实业务场景，更适合作为面试展示
