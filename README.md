# 比赛提交记录统计器

一个面向“竞赛管理开发岗”练手的小项目。

项目目标是模拟比赛后台常见的数据处理工作：输入一批提交记录，统计每题提交数、通过数、首次通过时间，以及每队提交数、通过题数等信息。

这个项目目前采用 `Python + CSV + 命令行` 的形式实现，重点放在数据建模、统计逻辑拆分和代码组织方式上，适合作为学习项目。

## 项目功能清单

### 已实现

- 读取 `CSV` 格式的提交记录
- 使用 `dataclass` 定义提交记录结构
- 统计每题提交数
- 统计每题通过数
- 统计每题首次通过时间
- 统计每队提交数
- 统计每队通过题数
- 按模块化函数组织读取、统计、输出逻辑

### 计划扩展

- 输出每题首个 `AC` 的队伍
- 同时展示题目名称和队伍名称
- 生成排行榜
- 增加罚时计算
- 按题目、队伍、状态进行筛选
- 导出统计结果为 `JSON`
- 增加异常数据校验
- 增加简单的 Web 展示页面

## 提交记录格式

当前使用 `CSV` 作为输入格式，表头如下：

```csv
submission_id,team_id,team_name,problem_id,problem_name,status,submit_time
```

字段说明：

- `submission_id`：提交编号
- `team_id`：队伍编号
- `team_name`：队伍名称
- `problem_id`：题目编号
- `problem_name`：题目名称
- `status`：提交结果
- `submit_time`：提交时间

其中：

- `status` 当前支持 `AC`、`WA`、`TLE`、`RE`、`CE`
- `submit_time` 格式为 `YYYY-MM-DD HH:MM:SS`

样例数据位于 `sample_data/submissions.csv`。

## 实现思路

这个项目不是一开始就直接写“大而全”的统计程序，而是按照“先读数据，再做最小统计，再逐步扩展”的思路来完成。

整体实现可以拆成 3 层：

1. 数据读取层

- 从 `CSV` 中逐行读取提交记录
- 将每一行转换成结构化的 `Submission` 对象

2. 业务统计层

- 按题目统计提交次数
- 按题目统计通过次数
- 按时间顺序统计首次通过时间
- 按队伍统计提交次数
- 利用 `set` 做去重，统计每队通过题数

3. 结果展示层

- 将零散统计结果统一汇总
- 用更清晰的格式输出到命令行

## 教学式开发步骤

为了方便学习，这个项目按下面的步骤手写：

### 第一步：定义数据结构

先使用 `@dataclass` 定义一条提交记录：

```python
@dataclass
class Submission:
    submission_id: str
    team_id: str
    team_name: str
    problem_id: str
    problem_name: str
    status: str
    submit_time: datetime
```

这样做的目的，是让每条数据从“字典”变成“结构清晰的对象”，后续访问字段时更直观。

### 第二步：读取 CSV

使用 `csv.DictReader` 逐行读取数据，并用 `datetime.strptime()` 将时间字符串转换为时间对象。

这一层的重点是：

- 先把输入数据读对
- 先确认每条记录都能正常解析
- 不急着一开始就写复杂统计

### 第三步：先写最简单的统计

从“每题提交数”开始：

- 遍历所有提交
- 使用 `problem_id` 作为字典键
- 每遇到一条记录就加 1

这是整个项目最基础的统计模型。

### 第四步：加条件过滤

再扩展到“每题通过数”：

- 只统计 `status == "AC"` 的记录
- 其余记录直接跳过

这一步体现的是“先过滤，再统计”的思路。

### 第五步：加时间维度

实现“每题首次通过时间”：

- 先按 `submit_time` 排序
- 再遍历记录
- 第一次遇到某题的 `AC` 时，记录时间

这一步体现的是“首次”这种业务需求一定依赖时间顺序。

### 第六步：按队伍统计

实现“每队提交数”和“每队通过题数”：

- 每队提交数：和每题提交数的写法类似，只是把键换成 `team_id`
- 每队通过题数：不能直接统计 `AC` 次数，因为同一队伍同一题可能多次通过

因此这里使用 `set` 去重：

- `set` 可以自动去掉重复题目编号
- 最后通过 `len(set)` 得到真正的通过题数

## 当前代码结构

当前主程序放在 `main.py`，主要分成几类函数：

- `load_submissions()`：读取提交记录
- `count_problem_submissions()`：统计每题提交数
- `count_problem_accepted()`：统计每题通过数
- `count_first_ac_time()`：统计每题首次通过时间
- `count_team_submissions()`：统计每队提交数
- `count_team_solved_problems()`：统计每队通过题数
- `analyze_submissions()`：汇总统计结果
- `print_problem_stats()` / `print_team_stats()`：输出结果

这种拆分方式的好处是：

- 每个函数只做一件事
- 逻辑更清晰
- 更方便后续增加新功能

在项目根目录执行：

```bash
python main.py sample_data/submissions.csv
```

## 这个项目适合面试时怎么讲

如果把它作为面试项目，可以重点讲下面几点：

- 为什么先选 `CSV` 而不是数据库
  适合快速验证业务逻辑，降低项目起步成本

- 为什么使用 `dataclass`
  让数据结构更清晰，代码可读性更好

- 为什么先从小统计开始
  先保证基础功能正确，再逐步扩展复杂业务

- 为什么统计“通过题数”时要用 `set`
  因为需要去重，不能把同一题的多次 `AC` 重复计数

- 为什么“首次通过时间”要先排序
  因为首次本质上是一个时间顺序问题
